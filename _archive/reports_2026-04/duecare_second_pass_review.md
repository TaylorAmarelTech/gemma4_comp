# DueCare Second-Pass Review

## 1. Executive one-pager

Verdict: DueCare is real software, not a pitch deck, but the repo currently overstates end-to-end readiness. The architecture, package split, workflow schema, demo API, CI, and Kaggle artifacts are substantive; the flagship one-line demo path is not yet truthful enough for judge scrutiny. (`pyproject.toml:23-35`; `packages/duecare-llm-core/src/duecare/core/__init__.py:1-3`; `packages/duecare-llm-workflows/src/duecare/workflows/__init__.py:1-12`; `.github/workflows/ci.yml:35-98`; `tests/integration/test_demo_app.py:18-117`; `kaggle/kernels/duecare_22_gemma_generations/kernel-metadata.json:2-24`; `kaggle/kernels/duecare_phase3_finetune/kernel-metadata.json:2-18`)

Best news: the technical substrate is credible. `duecare.core` is a proper contracts/schemas/provenance layer; `duecare.models`, `duecare.domains`, `duecare.tasks`, and `duecare.agents` are all actual registries with real shipped implementations; CI runs tests, builds all 8 wheels, performs a clean-room install, and executes 4 notebooks. (`packages/duecare-llm-core/src/duecare/core/__init__.py:1-77`; `packages/duecare-llm-models/src/duecare/models/__init__.py:1-24`; `packages/duecare-llm-domains/src/duecare/domains/__init__.py:1-43`; `packages/duecare-llm-tasks/src/duecare/tasks/__init__.py:1-25`; `packages/duecare-llm-agents/src/duecare/agents/__init__.py:1-27`; `.github/workflows/ci.yml:35-98`)

Worst news: the exact command the repo uses as its public story is broken in two ways. The packaged console script is `forge`, not `duecare`; and the CLI passes only a target-model id string into the workflow runner, while the Judge explicitly skips if no `target_model_instance` exists. The workflow smoke test treats that skipped Judge as a successful completed run. If a judge types the command from the README, judge guide, or video script, they will not get the evaluation story the docs promise. (`packages/duecare-llm/pyproject.toml:53-54`; `packages/duecare-llm/src/duecare/cli/cli.py:69`; `packages/duecare-llm-agents/src/duecare/agents/judge/judge.py:48-49`; `packages/duecare-llm-workflows/tests/test_workflows_smoke.py:94-96`; `README.md:115`; `docs/FOR_JUDGES.md:84-86`; `docs/video_script.md:79`)

Second-worst news: the workflow layer is better than the prompt implied, but it is still not trustworthy as the submission's central orchestrator. It has a real Pydantic workflow schema, real DAG validation, real runner code, and real smoke tests; but `WorkflowRun` advertises `agent_outputs`, `final_metrics`, and `final_artifacts` while the runner returns only `ctx.metrics` and `ctx.artifacts`, and the supervisor never merges agent `metrics`, `context_updates`, or `artifacts_written` into those top-level context fields. That means the orchestrator can execute agents and still emit a thin run record. (`packages/duecare-llm-workflows/src/duecare/workflows/loader/loader.py:12-45`; `packages/duecare-llm-workflows/src/duecare/workflows/dag/dag.py:9-28`; `packages/duecare-llm-workflows/tests/test_workflows_smoke.py:44-111`; `packages/duecare-llm-core/src/duecare/core/schemas/workflow.py:26-28`; `packages/duecare-llm-workflows/src/duecare/workflows/runner/runner.py:77-120`; `packages/duecare-llm-core/src/duecare/core/schemas/agent.py:52-54`; `packages/duecare-llm-agents/src/duecare/agents/base/base.py:156-170`)

Third: docs are drifting faster than code. `docs/EXTENDING.md` documents workflow YAML with `stages` and `requires`, but the loader only accepts `agents` and `needs`; the notebook count is 18 in `project_status`, 19 in the writeup, and 23 in `FOR_JUDGES`; and the trafficking domain metadata still says 12 seed prompts while runtime reports 74,567. Those are not cosmetic inconsistencies; they directly damage judge trust. (`docs/EXTENDING.md:306-332`; `packages/duecare-llm-workflows/src/duecare/workflows/loader/loader.py:12-45`; `docs/project_status.md:72`; `docs/writeup_draft.md:26`; `docs/FOR_JUDGES.md:163`; `configs/duecare/domains/trafficking/card.yaml:14`; `packages/duecare-llm-domains/src/duecare/domains/_data/trafficking/card.yaml:14`; `packages/duecare-llm-domains/src/duecare/domains/_data/trafficking/README.md:12`; `reports/test_run_001.md:12-24`)

Submission read: Phase 1 and Phase 4 are materially present, Phase 2 is half-present, Phase 3 is real but incomplete, and the wrap-up artifacts are still partly aspirational. The right move is not more breadth. It is to make the primary command, primary docs, and primary publication surfaces completely truthful. (`docs/project_phases.md:19-24`; `docs/FOR_JUDGES.md:163`; `docs/FOR_JUDGES.md:192-201`; `tests/integration/test_demo_app.py:18-117`; `kaggle/kernels/duecare_phase3_finetune/kernel-metadata.json:2-18`)

## 3. Glossary

### DueCare-specific terms

- **Domain pack**: a self-contained folder of `card.yaml`, `taxonomy.yaml`, `rubric.yaml`, `pii_spec.yaml`, `seed_prompts.jsonl`, and `evidence.jsonl`, loaded as data rather than code. (`packages/duecare-llm-domains/src/duecare/domains/__init__.py:3-6`; `packages/duecare-llm-domains/src/duecare/domains/pack/file_domain_pack.py:17-29`)
- **AgentSupervisor**: the meta-agent wrapper that applies retries, budget caps, and abort-on-harm around every agent call. (`packages/duecare-llm-agents/src/duecare/agents/__init__.py:22-27`; `packages/duecare-llm-agents/src/duecare/agents/base/base.py:89-117`)
- **WorkflowRunner**: the current YAML DAG executor that topologically orders agents and routes them through `AgentSupervisor`. (`packages/duecare-llm-workflows/src/duecare/workflows/__init__.py:1-12`; `packages/duecare-llm-workflows/src/duecare/workflows/runner/runner.py:17-120`)
- **`rapid_probe`**: the 5-minute smoke workflow for one model and one domain, currently defined as `scout -> judge -> historian`. (`configs/duecare/workflows/rapid_probe.yaml:1-14`)
- **Six-dimension judge**: the notebook-side 0-100 grading surface covering refusal, legal accuracy, completeness, victim safety, cultural sensitivity, and actionability. (`docs/notebook_guide.md:100`; `docs/FOR_JUDGES.md:123`)
- **Trafficking five-grade scale**: the repo's canonical response grading ladder is `worst`, `bad`, `neutral`, `good`, `best`, mapped to 0.0 through 1.0. (`configs/duecare/domains/trafficking/rubric.yaml:4-10`)
- **Business-framed exploitation**: exploitation requests disguised as professional or business inquiries that should still trigger trafficking detection. (`configs/duecare/domains/trafficking/rubrics/business_framed_exploitation.yaml:8-12`)
- **Jurisdictional hierarchy exploitation**: cross-border schemes that look legal in isolation but become trafficking when combined across jurisdictions. (`configs/duecare/domains/trafficking/rubrics/jurisdictional_hierarchy.yaml:8-11`)
- **Financial crime blindness**: failures to recognize predatory lending, laundering, and similar financial structures that exploit migrant workers. (`configs/duecare/domains/trafficking/rubrics/financial_crime_blindness.yaml:8-11`)
- **Prompt injection amplification**: adversarial formatting or jailbreak attempts that try to bypass safety detection without changing the underlying trafficking scheme. (`configs/duecare/domains/trafficking/rubrics/prompt_injection_amplification.yaml:8-11`)
- **Victim re-victimization**: cases where the model fails to validate or protect a potential trafficking victim who is seeking help. (`configs/duecare/domains/trafficking/rubrics/victim_revictimization.yaml:8-11`)
- **Migration corridor**: the structured origin-destination dimension used across prompts, schemas, and corridor registries. (`packages/duecare-llm-core/src/duecare/core/schemas/conversation.py:41`; `configs/duecare/corridors.yaml:1`)
- **Composite character**: a deliberately non-real named persona such as Maria, used for storytelling without putting real victim identities in the repo or video. (`docs/video_script.md:22`; `docs/video_script.md:46`; `.claude/rules/10_safety_gate.md:57-59`)

### External tech and repo-level terminology

- **Unsloth**: the planned fine-tuning stack for Phase 3. The repo consistently frames the fine-tuner as `Unsloth + LoRA on Gemma 4 E4B`. (`CLAUDE.md:119`; `docs/project_phases.md:23`)
- **LoRA**: the low-rank adaptation path the repo intends to train, then merge into export formats. (`CLAUDE.md:119-121`)
- **GGUF / llama.cpp**: the local-export target for post-fine-tune weights and the desktop deployment story. (`CLAUDE.md:121`; `CLAUDE.md:276`; `README.md:70`)
- **LiteRT**: the on-device mobile/runtime stretch target paired with GGUF in the export story. (`CLAUDE.md:121`; `CLAUDE.md:151`; `CLAUDE.md:292-293`)
- **E2B / E4B**: the two Gemma 4 sizes the repo keeps comparing, with E2B framed as smaller and E4B framed as higher quality. (`CLAUDE.md:290-291`; `README.md:101`; `configs/duecare/models.yaml:6-17`)
- **PEP 420 namespace packages**: the reason all 8 wheels share the `duecare` Python namespace. (`README.md:64`; `README.md:375`)
- **Pydantic v2**: the mandatory data-model system for schemas throughout the repo. (`CONTRIBUTING.md:65`; `CLAUDE.md:274`)
- **`typing.Protocol`**: the repo's preferred cross-layer contract mechanism instead of ABCs. (`README.md:371`; `CONTRIBUTING.md:67`)
- **Presidio**: the named PII-detection baseline used in the anonymization design. (`docs/architecture.md:866-868`)
- **FAISS**: the planned vector index for Phase 3 RAG retrieval. (`docs/project_phases.md:278`; `docs/project_phases.md:358`; `docs/project_phases.md:613`)
- **Ollama**: the local runtime used for dev-time Gemma execution and some comparison notebooks. (`README.md:91-101`; `README.md:186`)
- **jupytext**: the synchronization mechanism the publishing strategy wants between Kaggle notebooks and `.py` source-of-truth files. (`.claude/rules/50_publish_strategy.md:99`)
- **uv workspace**: the monorepo package-management model used at the repo root. (`pyproject.toml:23-35`)
