Background, literature & ethical framework

Why human exploitation and abuse warrants a dedicated tool, what prior work exists, what gap Duecare fills, and how the system should and should not be deployed.

What "trafficking" means here, technically

The Palermo Protocol Article 3 (UN, 2000) defines trafficking as the act (recruitment, transport, harbouring) + means (force, fraud, coercion, abuse of vulnerability, debt bondage, document retention) + purpose (exploitation, including forced labour). All three elements must be present. Duecare's heuristic + KB targets the means element because that's where the language patterns are most identifiable in recruitment ads, chats, and contracts.

The 11 ILO indicators of forced labour

The ILO Indicators of Forced Labour (2012) framework lists 11 operational indicators. Duecare's identify_ilo_indicators tool maps free-form input to these via curated regex:

  1. Abuse of vulnerability
  2. Deception
  3. Restriction of movement
  4. Isolation
  5. Physical and sexual violence
  6. Intimidation and threats
  7. Retention of identity documents ← the headline indicator for migrant domestic workers
  8. Withholding of wages
  9. Debt bondage ← the headline indicator for recruitment-fee schemes
  10. Abusive working and living conditions
  11. Excessive overtime

The four indicators above in bold are the most common patterns in recruitment-stage ads and chats — they're the ones Duecare's prescan is tuned for.

2. The current detection gap

There are good systems for parts of this problem, but none for the specific job we're targeting (real-time, per-case, flexible-deployment, across multiple jurisdictions and languages).

Commercial AML / KYC products Sanctions and PEP screening for banks. Built around financial entities, not recruitment language. ~$50K/year licensing — out of reach for the NGOs that need it.

The specific gap Duecare fills

A multilingual, real-time pre-filter that understands forced-labour and exploitation language patterns AND can cite the specific statute / convention / hotline a reviewer needs to act on it. Open-weight (Gemma 4 + Apache 2.0), Apache-2.0 runtime, deployment-mode-flexible (enterprise GPU batch or consumer / mobile via LiteRT), and built so that every verdict is auditable.

3. Technical literature this builds on

Retrieval-augmented generation (RAG)

The two-stage retrieve-then-generate pattern was formalized in Lewis et al., "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks", NeurIPS 2020 (arXiv:2005.11401). Duecare's stage 3 is a small-corpus instantiation: sentence-transformers/all-MiniLM-L6-v2 over 27 KB passages, top-K cosine. The KB is small because the legal vocabulary for trafficking is finite (statutes, conventions, indicators), so dense retrieval is overkill — but it covers paraphrase / multilingual cases that GREP misses.

Tool-augmented language models

The pattern of LLMs calling deterministic tools to compensate for brittle factual recall is best documented in Schick et al., "Toolformer: Language Models Can Teach Themselves to Use Tools", NeurIPS 2023 (arXiv:2302.04761). Gemma 4 ships native function-calling support in its chat template, so the path from prompt → tool call → result → final answer is single-model. Duecare's stage 4 demonstrates this with 7 deterministic tools (statute lookup, hotline, agency licence verification, fee-cap calculator, ILO indicator matcher, embassy lookup, known-actor search).

Flexible deployment (server, consumer, mobile)

The viability of running a 4–8 B parameter LLM on consumer hardware was established by the llama.cpp project (github.com/ggerganov/llama.cpp) and the GGUF + 4-bit quantization stack. Google's LiteRT (formerly TFLite) extends this to mobile with CPU + NNAPI + GPU delegates. Duecare targets the full spectrum: the FastAPI server uses the transformers library with bitsandbytes 4-bit quantization (~5.5 GB VRAM on a Kaggle T4) for enterprise + GPU batch; the consumer path runs on a single GPU; the mobile path targets LiteRT for sub-1 GB inference. The same KB, tools, and audit-trail format are shared across all three.

Multilingual + low-resource NLP

Migrant workers communicate in Tagalog, Bahasa Indonesia, Hindi, Bengali, Sinhala, Nepali, Urdu, Vietnamese, and Burmese — many of which are low-resource for English-trained models. Duecare's embedding model (MiniLM) covers these to a working degree because it was trained on the parallel OPUS + FLORES-200 corpora. Gemma 4 itself is multilingual via its training mix. For high-stakes deployment we recommend per-language fine-tuning (the Phase 3 Unsloth pipeline in this repo provides the scaffolding).

Trafficking-specific NLP work

Academic NLP work on trafficking is dominated by sex-trafficking detection in online ads (e.g., the TraffickCam project at Washington University and the HotelsCombined dataset). The most-cited paper is Tong et al., "Combating Human Trafficking with Multimodal Deep Models", ACL 2017 (aclanthology.org/P17-1138). Labour-trafficking detection (recruitment-fee fraud, kafala-system abuse) is comparatively underserved in the literature; the operational frameworks come from ILO, IOM, and Verité reports rather than peer-reviewed papers. Duecare positions itself in that operational gap.

The exploitation-LLM benchmark this work builds on

The submission's training and evaluation data is bootstrapped from Taylor Amarel's 21K-test human-exploitation benchmark (released as the public trafficking-llm-benchmark-gitlab repository). It contains graded response examples (5-point scale: harmful → incomplete → adequate → good → best) across the ILO indicator set, with scenarios spanning the major recruitment corridors (PH→GCC, ID→ME, NP→GCC, BD→ME, etc.) and the major operational dimensions (fee shifting, document retention, kafala transfer fraud, debt-bondage lending, dual-contract novation, family-vendor bundles, premium-credit platforms, cross-border novation).

4. Why Gemma 4 specifically

Three Gemma 4 features make this submission tractable:

  1. Native multimodal understanding. Gemma 4's E4B and E2B instruction-tuned variants accept image + text in a single forward pass via the AutoModelForImageTextToText class. This means a worker can paste a photo of a contract page and get a verdict on the same model — no separate OCR + vision encoder pipeline needed. (We currently use a text-only fallback for the demo because the multimodal Processor class isn't registered in transformers 4.57.x; the multimodal path is wired for transformers ≥ 5.5.0.dev0.)
  2. Native function calling. Gemma 4's chat template accepts a tools list parameter and emits structured tool-call JSON in a multi-turn loop. This is the substrate of Duecare's stage 4 — for the demo we use deterministic tool calls for reproducibility, but the agentic path is one bool flag away (see raw_python/gemma4_*_v1.py for the full Gemma-driven tool-calling agent loop).
  3. On-device deployable. Gemma 4 E4B at 4-bit quantization fits in ~5.5 GB VRAM (a Kaggle T4 has 15 GB). E2B fits in ~2 GB and runs on a high-end phone via LiteRT. For NGOs working with trafficking victims, this is the difference between "cannot use AI tools" and "deploys to a $300 laptop." The Apache 2.0 licence means no commercial restriction on humanitarian deployment.

5. Data-handling framework

Duecare can be deployed in any of three modes (enterprise GPU batch, single-operator on a workstation, or on-device via LiteRT) — and the data-handling rules below apply regardless of the deployment mode. The threat model isn't curiosity; it's revictimization:

  • A trafficked worker's case file contains the names of family members still in the origin country who could be retaliated against.
  • A NGO's caseload is a target for state actors hostile to labour-rights activism (Cambodia, GCC monarchies, etc.).
  • A platform's content-moderation queue is competitive intelligence (which agencies are getting flagged) that recruiters will exfiltrate if they can.

The architectural commitments that follow:

  1. Operator-controlled inference. Every Gemma 4 forward pass runs on hardware the operator chooses (enterprise GPU, single workstation, or mobile device). The default deployment does not send worker text to any external API.
  2. No logging of inputs. The structured event log (/logs) records request paths, status codes, durations, task lifecycle — but not the input text. The trace records what each stage found (a count, an ID), not the original content.
  3. Anonymizer gate. Anything that does end up in the evidence DB or training pipeline goes through the duecare.agents.anonymizer hard gate, which replaces names / passport numbers / phone numbers / addresses / financial accounts with tagged placeholders and stores only sha256 hashes of the originals.
  4. Composite characters in all examples. Maria, Sita, and Ramesh in the example set are composites synthesized from the public ILO + Verité case typology, not real victims.
  5. Open-source. Apache 2.0 model + MIT code so NGOs can audit every line.

6. Limitations and ethical use

What Duecare is not

  • Not a replacement for a human caseworker. Every BLOCK / REVIEW verdict is a triage signal, not a final determination. The architecture is designed to pass actionable evidence (KB passages, statute citations, hotline numbers) to a human, not to make the decision instead of one.
  • Not a sanctions or KYC product. The search_known_actors tool ships with a small mocked list for demonstration. Operators who deploy this should wire in a real registry (national POEA / DMW / EAA / MOM / MOHRE feeds; UN sanctions list; OFAC SDN).
  • Not a court-admissible determination. The system can surface evidence; it cannot establish guilt under any jurisdiction's evidentiary standards.

Known false-positive risks

  • Legitimate fee mentions — a POEA-licensed agency clearly stating their cap-compliant placement fee can fire the prescan. Mitigated by the legitimate-context signals (poea_license_mention, employer_pays_principle) and the licence-check tool, but operators should not auto-block on severity 4–6.
  • Cultural framing of "live-in" arrangements — some regions treat live-in domestic work as the default. The system flags it as a risk pattern; the human reviewer applies the cultural context.
  • Multilingual coverage is uneven. English / Tagalog / Indonesian are well-covered; Bengali / Sinhala / Nepali / Urdu rely on MiniLM's cross-lingual embeddings and may miss idiomatic local phrasings.

Recommended deployment posture

Duecare should be deployed as a pre-filter to a human-review queue, not as an automatic enforcement engine. The output of a moderate run is structured evidence designed to make a human's review faster and better-informed — citation, hotline, embassy contact, and explicit ILO indicator matches all in one record. Auto-blocking should be reserved for severity-9-or-10 with multiple critical signals AND no legitimate-context signals.

8. Reproducibility

Every numeric claim in the writeup and demo is traceable to (git_sha, dataset_version). The code repository is public; the wheels are versioned on PyPI and Kaggle Datasets; the fine-tuned model weights are tagged and pushed to Hugging Face Hub (Phase 3). To reproduce a run: 

# Local
pip install duecare-llm
duecare run --input ./case_files/ --output ./out/

# Or via the Kaggle notebook (one-click)
# https://www.kaggle.com/code/taylorsamarel/duecare-live-demo

The exact severity / verdict any input produces is deterministic given (model, KB version, heuristic rules). The trace exposed via /queue records every stage's output so a researcher or auditor can verify the decision path.
